## Notas de la clase

| Escalabilidad | Vertical | Horizontal |
| - | :-: | :-: |
|  | Agrega más recursos al Servidor físicamente | Agrega más equipos Servidores |
|  | Existe una limitación física | No existe una limitación física |
|  | No resuelve el problema de Disponibilidad, sino que sólo resuelve el problema de Performance | Resuelve el problema de Disponibilidad y de Performance |
|  |  | Conlleva una complejidad extra ya que se debe distribuir la carga entre todos los servidores |
|  |  | Es necesario contar con un Balanceador de Cargas |

* La escalabilidad horizontal no siempre es la mejor solucion por defecto.

| Aplicativo | Monolítico | No Monolítico / Distribuido |
| - | :-: | :-: |
|  | Tienen toda la lógica de negocio en un único proyecto de código fuente | Se compone de múltiples componentes de software o servicios que se ejecutan en diferentes nodos, pero que cooperan para ofrecer una funcionalidad unificada |
|  | Si se realiza un cambio en el código fuente, todo el aplicativo debe desplegarse por completo | Se permiten actualizaciones y despliegues independientes por cada nodo |

* Service Oriented Architecture (SOA): Es un estilo de arquitectura donde se descompone la estructura funcional en varios servicios. Está compuesto por:
  * Consumidor de Servicios: Es una aplicación, módulo de software u otro servicio que demanda la funcionalidad proporcionada por un servicio.
  * Proveedor de Servicios: 
  * Registro de Servicios:
  * Bus de Servicio Empresarial (ESB): 

* Balanceador de Cargas: Distribuye las solicitudes entrantes entre múltiples servidores o instancias de aplicación para optimizar el rendimiento y evitar sobrecargas.

* Caché: Almacena resultados de operaciones o datos de uso frecuente para reducir el tiempo de acceso y la carga sobre los servicios principales o las bases de datos.
