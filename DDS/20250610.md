## Notas de la clase

Patrones arquitectónicos de Solicitud - Respuesta: Engloban patrones de arquitectónicos con un estilo de interacción sincrónica donde un cliente envía una solicitud a un servidor y espera una respuesta. Se caracterizan por tener un acoplamiento fuerte entre cliente y servidor, con una comunicación directa y sin intermediarios.

Patrón Subcripción de Eventos: Un sujeto (emisor) notifica a múltiples observadores (subcriptores) cuando ocurre un evento. Los observadores se registran para recibir notificaciones.

Caracteristicas:
  * Leve desacoplamiento
  * Notificación automática sin "polling"
  * Uno a muchos
Limitaciones:
  * Aún existe acoplamiento entre el emisor y los subcriptores 
  * No escala bien si necesita persistencia

Patrón Publicación - Subscripción: Es un patrón de mensajería donde se definen publicadores y subcriptores, que no se conocen entre sí. Los subcriptores se suscriben a un tópico, en el cual el publicador genera sus mensajes para éste.

Características:
  * Descoplamiento total
  * Alta escalabilidad
  * Distribución natural
  * Asincronía
  * Alta extensibilidad
Limitaciones:
  * Sin persistencia de mensajes
  * Sin control de calidad de servicio, orden o seguridad

WebSockets: Para aplicar una arquitectura Publisher - Subscriber en tiempo real en una aplicación web, se suele usar *WebSocket* como canal de comunicación. Los WebSocket permiten que el navegador establezca una conexión persistente con el backend.

Broker: Es un patrón arquitectónico que consiste en desacoplar emisores y receptores mediante un intermediario inteligente (tiene lógica propia). Es la forma más común de implementar el patron Publisher - Subscriber, permitiendo ese desacoplamiento total. El broker se encarga de recibir, almacenar, filtrar y distribuir los mensajes entre publishers y subcribers

Características:
  * Desacoplamiento total
  * Persistencia
  * Alta escalabilidad
  * Enrutamiento inteligente
  * QoS (Calidad de Servicio)
  * Seguridad
  * Monitorización en tiempo real

Clase producto con una descripcion (String), precio base (Double) y un tipo (clase TipoDeProducto).

El TipoDeProducto contiene su descripcion y los impuestos (lista de Impuesto). Calcula tambien el total del los impuestos para un producto en particular.

Es la solución con más extensibilidad, ya que si se quiere agregar un nuevo tipo de producto simplemente se instancia otro TipoDeProducto.

Se crea una interfaz impuesto con una función para calcular los impuestos para un producto. Luego se generan clases concretas para cada tipo de impuesto a calcular. Cumple con Sustitución Liskov, Abierto - Cerrado y Responsabilidad Única, manteniendo alta cohesión y mantenibilidad.

La integración de la clase oficina contable con el sistema contable se hace por medio de un adapter que se encarga de simplificar la implementación de la notificación de venta de un producto. Permite también el testeo por medio de mocking.
